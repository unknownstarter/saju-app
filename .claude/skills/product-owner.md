# Product Owner — 사주 데이팅 앱

## 1. 사용자 스토리 작성 가이드

### 스토리 포맷
```
[사용자 유형]으로서, [목표/행동]을 하고 싶다. 왜냐하면 [이유/가치]이기 때문이다.

인수 조건:
- [ ] 조건 1
- [ ] 조건 2
- [ ] 조건 3
```

### 사용자 유형 (Persona)
| 페르소나 | 설명 | 핵심 니즈 |
|---------|------|----------|
| 사주 입문자 | 사주에 관심은 있지만 잘 모르는 2030 | 쉽고 재미있는 사주 해석 |
| 사주 매니아 | 이미 사주를 알고 있고 궁합을 중시하는 사용자 | 정확한 사주 분석과 궁합 매칭 |
| 일반 데이팅 유저 | 사주보다 일반적인 데이팅 기능을 원하는 사용자 | 좋은 매칭, 안전한 대화 |
| 재미 추구형 | SNS 공유, 친구와 궁합 보기 등 엔터테인먼트 목적 | 바이럴 콘텐츠, 공유 기능 |

### 핵심 에픽 & 스토리 예시

#### Epic 1: 온보딩 & 프로필
```
사주 입문자로서, 생년월일시를 입력하면 내 사주 분석 결과를 바로 보고 싶다.
왜냐하면 앱을 깔자마자 가치를 느끼고 싶기 때문이다.

인수 조건:
- [ ] 양력/음력 선택 가능
- [ ] 생시를 모르는 경우 "모름" 옵션 제공
- [ ] 입력 후 3초 이내 사주 분석 결과 표시
- [ ] 오행 비율, 일간 성격, 올해 운세 요약 포함
- [ ] 결과 화면에서 "프로필 완성하기" CTA 제공
```

```
일반 데이팅 유저로서, 사진과 자기소개를 매력적으로 꾸미고 싶다.
왜냐하면 좋은 매칭을 받고 싶기 때문이다.

인수 조건:
- [ ] 사진 최소 1장, 최대 6장 업로드
- [ ] 자기소개 최대 500자
- [ ] 키, 직업, 위치, 음주/흡연 등 기본 정보 입력
- [ ] 사주 기반 성격 태그 자동 생성 (예: #불의열정 #나무의성장)
- [ ] 프로필 미리보기 기능
```

#### Epic 2: 사주 분석
```
사주 매니아로서, 내 사주팔자의 상세 분석을 보고 싶다.
왜냐하면 정확한 분석이 있어야 이 앱을 신뢰할 수 있기 때문이다.

인수 조건:
- [ ] 사주팔자 원국 표시 (연주, 월주, 일주, 시주)
- [ ] 오행 분포 차트 (목, 화, 토, 금, 수)
- [ ] 십신 관계 해석
- [ ] 대운/세운 타임라인
- [ ] AI 기반 자연어 해석 (쉬운 한국어)
```

#### Epic 3: 매칭
```
사주 입문자로서, 나와 사주 궁합이 좋은 사람을 추천받고 싶다.
왜냐하면 사주 궁합이 좋으면 실제로도 잘 맞을 것 같기 때문이다.

인수 조건:
- [ ] 매일 3-5명의 추천 (무료), 10명 (프리미엄)
- [ ] 각 추천에 궁합 점수 표시 (오행 기반)
- [ ] 궁합 점수의 근거 설명 (예: "두 분 모두 목(木) 기운이 강해 성장 지향적")
- [ ] 좋아요/패스 스와이프 기능
- [ ] 상호 좋아요 시 매칭 성사
```

#### Epic 4: 채팅
```
매칭된 사용자로서, 상대방과 대화를 시작하고 싶다.
왜냐하면 매칭 후 대화가 관계 발전의 핵심이기 때문이다.

인수 조건:
- [ ] 매칭 성사 시 채팅방 자동 생성
- [ ] 텍스트 메시지 실시간 전송/수신
- [ ] 사주 기반 대화 주제 추천 (아이스브레이커)
- [ ] 읽음 표시
- [ ] 신고/차단 기능
```

#### Epic 5: 구독 & 결제
```
활발한 사용자로서, 더 많은 매칭과 고급 사주 분석을 이용하고 싶다.
왜냐하면 무료 기능만으로는 부족하기 때문이다.

인수 조건:
- [ ] 무료/프리미엄/VIP 3단계 요금제
- [ ] 프리미엄: 무제한 좋아요, 상세 궁합 분석, 슈퍼좋아요 3회/일
- [ ] VIP: 프리미엄 + 누가 나를 좋아했는지 보기, AI 메시지 코칭
- [ ] Apple/Google 인앱결제
- [ ] 무료 체험 기간 (3일 또는 7일)
```

## 2. 우선순위 프레임워크 (RICE)

### RICE 점수 계산
```
RICE Score = (Reach × Impact × Confidence) / Effort
```

| 요소 | 설명 | 측정 방법 |
|-----|------|----------|
| **Reach** | 분기 내 영향받는 사용자 수 | 예상 사용자 수 (1000명 단위) |
| **Impact** | 개별 사용자에 대한 영향도 | 3=massive, 2=high, 1=medium, 0.5=low, 0.25=minimal |
| **Confidence** | 추정의 확신도 | 100%=high, 80%=medium, 50%=low |
| **Effort** | 필요한 개발 인력-주 | person-weeks |

### MVP 기능 우선순위 (예시)

| 기능 | Reach | Impact | Confidence | Effort | RICE | 우선순위 |
|-----|-------|--------|-----------|--------|------|---------|
| 소셜 로그인 (카카오) | 10 | 3 | 100% | 2 | 15.0 | P0 |
| 사주 입력 & 기본 분석 | 10 | 3 | 90% | 3 | 9.0 | P0 |
| 프로필 생성 | 10 | 2 | 100% | 3 | 6.7 | P0 |
| 기본 매칭 (궁합 기반) | 8 | 3 | 80% | 4 | 4.8 | P0 |
| 채팅 | 6 | 3 | 90% | 5 | 3.2 | P0 |
| 사주 카드 SNS 공유 | 8 | 2 | 70% | 1 | 11.2 | P1 |
| 상세 궁합 분석 | 5 | 2 | 70% | 3 | 2.3 | P1 |
| 인앱결제 (구독) | 3 | 3 | 80% | 4 | 1.8 | P1 |
| 오늘의 운세 (데일리) | 7 | 1 | 80% | 2 | 2.8 | P1 |
| AI 메시지 코칭 | 3 | 2 | 50% | 4 | 0.75 | P2 |
| 프로필 부스트 | 2 | 1 | 60% | 2 | 0.6 | P2 |

### 우선순위 레벨
- **P0 (Must Have)**: MVP 론칭에 필수. 이것 없으면 앱이 아님
- **P1 (Should Have)**: 론칭 후 1-2주 내 추가. 리텐션과 수익화에 핵심
- **P2 (Nice to Have)**: 데이터 보고 결정. 가설 검증 필요
- **P3 (Later)**: 백로그에 보관. 분기 계획 시 재검토

## 3. 스프린트 계획

### 스프린트 구조
- **스프린트 기간**: 1주 (빠른 반복, MVP 단계에 적합)
- **계획 미팅**: 월요일 30분
- **데일리**: 매일 15분 (비동기 가능)
- **리뷰 & 회고**: 금요일 30분

### 스프린트 벨로시티 추정 (1인 개발 기준)
- 1 Story Point = 약 2시간 작업
- 주당 가용 시간: 약 30시간 (집중 시간)
- 주당 벨로시티: 약 12-15 SP

### MVP 스프린트 로드맵 (예시)
```
Sprint 1: 프로젝트 셋업 & 인증
  - Flutter 프로젝트 구조 (3 SP)
  - Supabase 초기 셋업 (2 SP)
  - 소셜 로그인 - 카카오 (5 SP)
  - 소셜 로그인 - Apple (3 SP)

Sprint 2: 사주 분석 코어
  - 생년월일시 입력 UI (3 SP)
  - 만세력 기반 사주 계산 로직 (8 SP)
  - 사주 결과 화면 (5 SP)

Sprint 3: 프로필 & 사진
  - 프로필 입력 플로우 (5 SP)
  - 사진 업로드 & 크롭 (5 SP)
  - 프로필 카드 UI (3 SP)

Sprint 4: 매칭 시스템
  - 궁합 계산 알고리즘 (8 SP)
  - 매칭 추천 UI (5 SP)
  - 좋아요/패스 기능 (3 SP)

Sprint 5: 채팅
  - 채팅 UI (5 SP)
  - Supabase Realtime 연동 (5 SP)
  - 매칭 알림 (3 SP)

Sprint 6: 구독 & 결제
  - RevenueCat 연동 (5 SP)
  - 페이월 UI (3 SP)
  - 무료/프리미엄 기능 분기 (3 SP)

Sprint 7-8: 폴리싱 & 베타
  - 버그 수정 & 성능 최적화
  - TestFlight / 내부 테스트
  - 사주 카드 공유 기능
  - 온보딩 개선
```

## 4. 백로그 관리

### 백로그 구조
```
Epic
  └── Feature
       └── User Story
            └── Task (기술 작업)
```

### 백로그 상태 관리
| 상태 | 설명 |
|-----|------|
| Icebox | 아이디어 단계. 언젠가 할 수도 있는 것 |
| Backlog | 검증된 아이디어. 우선순위 정렬 대기 |
| Ready | 스토리 포인트 추정 완료. 스프린트 투입 가능 |
| In Progress | 현재 스프린트에서 진행 중 |
| Review | 코드 리뷰 / QA 중 |
| Done | 완료. 인수 조건 모두 충족 |

### Definition of Done (DoD)
- [ ] 인수 조건 100% 충족
- [ ] 단위 테스트 작성 (커버리지 80% 이상)
- [ ] UI 테스트 (위젯 테스트) 작성
- [ ] 코드 리뷰 완료
- [ ] 에러 핸들링 구현
- [ ] 로딩/빈 상태/에러 상태 UI 처리
- [ ] iOS & Android 양쪽 테스트
- [ ] 접근성 기본 체크 (semantic labels 등)

### 기술 부채 관리
- 기술 부채도 백로그 아이템으로 등록
- 매 스프린트 20% 용량을 기술 부채 해소에 할당
- 부채 심각도: Critical / High / Medium / Low
- Critical 부채는 다음 스프린트에 반드시 처리

## 5. 핵심 지표 (KPI)

### North Star Metric
**주간 매칭 성사 후 대화한 쌍 수 (Weekly Meaningful Matches)**
- 단순 매칭이 아닌 "대화까지 이어진 매칭"이 핵심 가치

### 퍼널 지표
```
설치 → 가입 → 프로필완성 → 첫매칭 → 첫메시지 → 구독
  100%  → 60%  →   40%    →  25%   →   15%   →  5%
  (목표) (목표)   (목표)    (목표)    (목표)   (목표)
```

### 주요 지표 대시보드

#### Engagement
| 지표 | 설명 | 목표 |
|-----|------|-----|
| DAU | 일간 활성 사용자 | 성장 추세 |
| MAU | 월간 활성 사용자 | 성장 추세 |
| DAU/MAU | 스티키니스 | > 20% |
| 세션 시간 | 평균 앱 사용 시간 | > 5분/세션 |
| 세션 빈도 | 일 평균 앱 오픈 횟수 | > 2회/일 |

#### Matching
| 지표 | 설명 | 목표 |
|-----|------|-----|
| 매칭률 | 좋아요 → 상호 매칭 비율 | > 5% |
| 대화 전환율 | 매칭 → 첫 메시지 | > 60% |
| 대화 지속률 | 5회 이상 메시지 교환 비율 | > 30% |
| 좋아요 일 평균 | 사용자당 일 좋아요 수 | 10-20회 |

#### Retention
| 지표 | 설명 | 목표 |
|-----|------|-----|
| D1 Retention | 다음 날 복귀율 | > 40% |
| D7 Retention | 7일 후 복귀율 | > 20% |
| D30 Retention | 30일 후 복귀율 | > 10% |
| 이탈 사유 | 이탈 사용자 설문 | 주기적 분석 |

#### Revenue
| 지표 | 설명 | 목표 |
|-----|------|-----|
| 구독 전환율 | 무료 → 유료 전환 | > 5% |
| ARPU | 사용자당 평균 매출 | 성장 추세 |
| ARPPU | 결제 사용자당 매출 | > ₩15,000/월 |
| LTV | 사용자 생애 가치 | > CAC × 3 |
| Churn Rate | 월간 구독 해지율 | < 10% |

#### Saju-specific
| 지표 | 설명 | 목표 |
|-----|------|-----|
| 사주 카드 공유율 | 사주 결과 SNS 공유 비율 | > 15% |
| 궁합 조회 수 | 일 평균 궁합 조회 | 성장 추세 |
| 오늘의 운세 조회율 | DAU 중 운세 확인 비율 | > 50% |
| 사주 기반 매칭 만족도 | 매칭 후 만족도 평가 | > 4.0/5.0 |

## 6. 사용자 여정 맵

### 전체 여정
```
인지 → 설치 → 온보딩 → 탐색 → 매칭 → 대화 → 관계발전 → (구독/이탈)
```

### 상세 여정 맵

#### Stage 1: 인지 & 설치
```
터치포인트: SNS 광고, 친구 공유, ASO, 입소문
감정: 호기심, 기대감
행동: 앱스토어 검색, 리뷰 확인, 다운로드
핵심 장벽: "또 다른 데이팅 앱?" → 사주라는 차별점으로 돌파
기회: 사주 카드 바이럴, "너 이거 해봤어?" 공유
```

#### Stage 2: 온보딩 (가장 중요!)
```
터치포인트: 앱 첫 실행
감정: "빨리 내 사주 결과 보고 싶다" → "오 재밌다!"
행동: 생년월일시 입력 → 사주 결과 확인 → 프로필 작성 → 사진 업로드
핵심 장벽:
  - 생시를 모르는 경우 → "모름" 옵션 + 부분 분석
  - 프로필 작성 피로 → 사주 기반 자동 태그 생성으로 부담 감소
  - 사진 업로드 거부 → "사진 없이 시작하기" 허용 (후에 넛지)
기회: 사주 결과의 WOW 모먼트가 가입 완료의 핵심 동력
```

#### Stage 3: 탐색 & 매칭
```
터치포인트: 홈 피드, 추천 카드
감정: 설렘, 기대, 때로는 실망
행동: 프로필 탐색, 좋아요/패스, 궁합 점수 확인
핵심 장벽:
  - 초기 사용자 부족 → 지역 확대, AI 프로필 대화 연습
  - 매칭 안 됨 → 좋아요 보낸 사람 힌트 제공 (프리미엄)
기회: 궁합 점수가 프로필 탐색의 동기 부여
```

#### Stage 4: 대화 & 관계 발전
```
터치포인트: 채팅방
감정: 긴장, 설렘, 재미
행동: 첫 메시지, 대화, 연락처 교환
핵심 장벽:
  - 첫 메시지 부담 → 사주 기반 아이스브레이커 제안
  - 대화 단절 → AI 대화 주제 추천
기회: "우리 궁합 이래요!" 대화 스타터
```

## 7. 사주 기능 vs 일반 데이팅 기능 밸런스

### 원칙
```
사주 = 차별화 & 바이럴 엔진
데이팅 = 핵심 가치 & 리텐션 엔진
둘 다 필요하지만, 사주가 "왜 이 앱이어야 하는가"의 답이다.
```

### 밸런스 매트릭스

| 기능 카테고리 | 비중 | 이유 |
|-------------|------|------|
| 사주 분석 & 궁합 | 30% | 차별화의 핵심. Unique Selling Point |
| 매칭 & 추천 | 25% | 데이팅 앱의 근본 가치 |
| 프로필 & 사진 | 20% | 매칭의 기본 재료 |
| 채팅 & 소통 | 15% | 매칭 후 관계 발전의 핵심 |
| 구독 & 수익화 | 10% | 비즈니스 지속성 |

### 의사결정 원칙
1. **사주가 데이팅을 방해하면 안 된다**: 사주를 몰라도 앱을 충분히 사용할 수 있어야 함
2. **사주가 데이팅을 강화해야 한다**: 궁합 점수가 매칭의 신뢰도를 높여야 함
3. **사주가 바이럴을 만들어야 한다**: SNS 공유 가능한 사주 콘텐츠가 성장 엔진
4. **과도한 사주 의존 금지**: "사주가 안 맞으니 만나지 마세요"는 절대 금지. 참고 자료일 뿐
